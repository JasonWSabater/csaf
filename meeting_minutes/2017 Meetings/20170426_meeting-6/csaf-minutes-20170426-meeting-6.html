6. Current Open JIRA Issues

6.1 CSAF-28 - Proposal for CVSS future embrace

Stefan walks through the proposal

All discuss the uniqueness constraint on the product id (D) in the proposal

Troy explains the uniqueness

Troy explains every vuln relates to only one product (so you do not have contradicting vulns per product

Troy suggests to enforce the constraints per versioned score set (thus allowing also v2 and v3 to target the same product id)

Omar understands, that we have to allow some flexibility in the cardinalities of the version scores.

Denny moves to allow 0 or 1 scrore sets per CVSS version. Harold seconds.

Omar: Unanimous consent, the motion carries

Feng suggests (based on the mail he sent) to change only minimally in v1.2 (and keep the private scale type for v2.0)

Stefan suggests to then reference cvssv3 for 0 - 10

All discuss the proposal in CSAF-28

Eric asks if the members of the TC think, the version 1.2 shall accomplish become a OASIS spec fast similar to 1.1 but supporting cvss v3 and thinks xml schema use to embed future compatibility is hard.

Troy accepts A.1 (from CSAF-28 proposal), but A.2 might be to far reaching

Harold is in favor to stay within narrow scope

Stefan I move to implement CSAF-28 but without the private type for cvssScoreType for the next working draft revision

Harold suggests to also remove D

Stefan I move to implement CSAF-28 but without the private type for cvssScoreType and without the uniqueness constraint (D) for the next working draft revision

All discuss

Stefan I move to implement CSAF-28 but without the private type for cvssScoreType and with the uniqueness constraint (D)changed, so it targets v2 and v3 scores separately thus will allow one unique v2 and one unique v3 score per product id for the next working draft revision

Feng suggests to instead not implement CSAF-28, but instead amend Feng V2 V3 explicit variant with an additional constraint to accommodate for v2/v3 uniqueness and Harold is OK

Harold suggests to relax the cardinalities on the v3 score set to allow for 0 occurrences, and to have upper bound of 1 instead of unbounded for v2 and v3

Feng suggests to keep unbounded

Harold concurs with that

Stefan I move to keep the existing schema as discussed above with two changes: One is adding a V2 constraint on productID and TWO is relax the minOccur ov ScoreSetV3 from 1 to 0. Vincent seconds.

Omar: No objections, unanimous consent the motion carries
